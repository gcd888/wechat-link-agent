## 1. 业务边界

### 1.1 项目定位

微连是一款 **Electron 桌面应用 + 微信机器人**，核心能力是将电脑上的 AI Agent CLI 工具（如 Claude Code、OpenCode、CodeBuddy 等）与微信打通，让用户通过微信远程操控这些 AI 编程助手。

### 1.2 系统边界图

```
┌─────────────────────────────────────────────────────────┐
│                      微连系统边界                         │
│                                                         │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │ 微信用户  │───▶│  微连桌面应用  │───▶│  AI Agent CLI │  │
│  │ (手机端)  │◀───│  (Electron)  │◀───│  (本地进程)    │  │
│  └──────────┘    └──────┬───────┘    └──────────────┘  │
│                         │                               │
│                  ┌──────┴───────┐                       │
│                  │  iLink Bot   │                       │
│                  │  API (微信)   │                       │
│                  └──────────────┘                       │
└─────────────────────────────────────────────────────────┘

外部依赖：
  - iLink Bot API（微信消息收发）
  - AI Agent CLI（Claude Code / OpenCode / CodeBuddy 等）
  - WebDAV 服务器（可选，数据同步）
```

### 1.3 核心价值链

| 环节 | 输入 | 处理 | 输出 |
|------|------|------|------|
| 微信接收 | 用户微信消息 | 长轮询 → 命令路由 | 消息/命令分发 |
| Agent 调用 | 用户指令 + 工作目录 | Provider 适配 → 子进程调用 | 流式输出 |
| 微信回复 | Agent 输出 | 长消息拆分 → API 回复 | 微信消息推送 |
| 数据持久化 | 消息/配置/会话 | SQLite 写入 | 本地数据库文件 |

## 2. MVP 范围

### 2.1 MVP 目标

> 让开发者通过微信发送消息给电脑上的 Claude Code，并实时收到流式回复。

### 2.2 In-Scope（当前版本做什么）

| 功能模块 | MVP 范围 | 优先级 |
|----------|----------|--------|
| Agent 管理 | PATH 自动扫描、Agent 列表展示、切换 | P0 |
| 微信集成 | 扫码登录、长轮询消息、命令路由、流式回复 | P0 |
| 会话管理 | 会话创建、切换、消息历史、导出 Markdown | P0 |
| 配置中心 | 主题/语言切换、工作目录设置、托盘最小化 | P0 |
| 数据安全 | API Key AES-GCM 加密、主密码保护 | P0 |
| Agent 安装 | 一键安装引导（npm/brew/winget） | P1 |
| WebDAV 同步 | 手动上传/下载、连接测试 | P1 |
| 工具箱 | LLM 供应商管理、API Key 加密存储 | P1 |
| 商城 | 合作工具展示、安装链接 | P1 |

### 2.3 Out-of-Scope（当前版本不做什么）

| 排除项 | 原因 | 后续计划 |
|--------|------|----------|
| 插件系统 | 当前聚焦核心功能稳定 | V2.0 规划 |
| 企业版 | 单用户场景优先 | V2.0 规划 |
| 开放 API | 先验证产品价值 | 生态化阶段 |
| 移动端 App | 微信已作为移动入口 | 无需独立 App |
| 多用户协作 | 桌面应用天然单用户 | 企业版考虑 |
| 语音/视频消息 | 文本交互已满足核心需求 | 后续评估 |
| 自动更新 | MVP 阶段手动下载更新 | V1.0 规划 |

## 3. 干系人分析

### 3.1 目标用户

| 用户画像 | 特征 | 核心诉求 |
|----------|------|----------|
| AI 开发者 | 已安装 Claude Code 等 CLI 工具 | 外出时远程操控 AI 编码 |
| 技术爱好者 | 喜欢尝试新 AI 工具 | 一个入口管理多种 Agent |
| 远程办公者 | 家里有高性能电脑 | 手机微信利用家里算力 |

### 3.2 使用场景

**场景 1：通勤路上审查代码**
- 用户在地铁上收到 PR 通知
- 通过微信发送消息给 Claude Code
- Claude Code 读取本地代码仓库，生成审查意见
- 流式回复到微信

**场景 2：多 Agent 切换**
- 用户同时安装了 Claude Code 和 OpenCode
- 通过 `/agent opencode` 切换到 OpenCode
- 发送不同的编程问题，获取不同 Agent 的建议

**场景 3：多设备数据同步**
- 公司电脑配置了 Agent 和 API Key
- 通过 WebDAV 同步到家里电脑
- 两台电脑共享会话历史和配置

## 4. 约束与假设

### 4.1 技术约束

- **微信接口**：依赖 iLink Bot API，非微信官方接口，存在稳定性风险
- **桌面应用**：必须安装并运行在用户电脑上，不能纯云端
- **CLI 依赖**：Agent CLI 必须已安装并配置好认证（如 `claude auth login`）
- **跨平台**：需同时支持 Windows、macOS、Linux

### 4.2 业务假设

- 用户具备基本的开发者技能（能安装 CLI 工具、配置环境变量）
- 用户拥有微信账号并愿意扫码登录
- 用户电脑上已安装 Node.js 运行时（Agent CLI 依赖）
- iLink Bot API 服务持续可用

### 4.3 合规约束

- 遵守微信使用规范，不做批量群发
- 用户数据仅存储在本地（除非主动开启 WebDAV 同步）
- API Key 加密存储，不明文传输
- 开源协议（MIT），允许社区贡献
